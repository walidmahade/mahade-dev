Webflow SEO is not complicated. The platform handles a lot of the fundamentals — clean HTML output, global CDN, automatic HTTPS, sitemap generation — better than most WordPress setups do out of the box. The problems I see on sites that aren't ranking are almost always in the 20% of decisions the developer or site owner still has to make deliberately: heading hierarchy, page titles, canonical setup, schema markup, and image optimization.

This checklist covers all of it. I've built 40+ Webflow sites across SaaS, agency, e-commerce, and professional services. These are the exact items I run through on every project before and after launch — 60+ checks organized by category so you can work through them without doubling back. If you want to understand where Webflow sits relative to WordPress on these fundamentals, I wrote a direct comparison in [Webflow vs WordPress 2026](/blog/webflow-vs-wordpress-2026).

## The 5-minute Webflow SEO sanity check (run this first)

Before going deep on any section, do a fast pass on the five items that cause the most indexing failures. These are the checks that, if wrong, mean nothing else on this list matters — because Google either isn't crawling your site or is crawling the wrong version of it.

| Check | How | Pass / Fail indicator |
|---|---|---|
| Indexing not blocked | Webflow Dashboard → SEO → "Disable Webflow subdomain indexing" is OFF; Page Settings → "Exclude from search" is OFF on live pages | Fail: Googlebot can't reach your content |
| Sitemap is live | Visit `yourdomain.com/sitemap.xml` directly | Fail: 404 means auto-sitemap is off or custom domain isn't connected |
| Search Console verified | Google Search Console → Coverage report shows pages indexed | Fail: "URL is not on Google" for your homepage |
| Canonical points to live domain | View source → find `<link rel="canonical">` — should be `https://yourdomain.com/page-slug/` not `yoursite.webflow.io` | Fail: staging subdomain in canonical leaks link equity |
| HTTPS enforced | Visit `http://yourdomain.com` — should redirect to `https://` | Fail: mixed content or no redirect splits ranking signals |

Any single failure here is a blocker. Fix it before working through the rest of the list. The most common silent killer is a staging subdomain that a developer forgot to exclude — Googlebot finds it, indexes it, and then your live site looks like duplicate content.

## On-page Webflow SEO: titles, meta, H1-H3, URLs

This is the layer most people think they've covered when they haven't. Setting a page title is not the same as setting a good page title. Here's what actually matters.

| Field | Where in Webflow | Best practice |
|---|---|---|
| Page title | Page Settings → SEO tab → Title Tag | 50–60 characters, primary keyword near the front, brand name at the end |
| Meta description | Page Settings → SEO tab → Meta Description | 140–160 characters, reads like an ad, includes the keyword once |
| H1 | Webflow Designer → first text element on page | One per page, contains the primary keyword, matches search intent |
| URL slug | Page Settings → Slug | Lowercase, hyphens only, keyword-forward, no dates (for evergreen content) |
| Open Graph image | Page Settings → Open Graph → OG Image | 1200×630px, exported as JPG under 200KB |

<Callout type="tip">
Webflow's global title template (Site Settings → SEO → Title Tag) lets you set a suffix like ` | Your Brand` that appends to every page title. Use it — but write the first part of every page title manually. Auto-generated titles from CMS slugs or page names are almost never the right keyword targets.
</Callout>

**H1 is the field I see wrong most often.** Webflow doesn't enforce a single H1 — you can drop five of them on a page and the Designer won't warn you. On every audit I run, I find homepages with an H1 in the hero, an H1 on the features grid, and an H1 on the pricing section. Google's John Mueller has said multiple H1s aren't a direct penalty, but a single focused H1 is still best practice because it signals clearly what the page is about.

**URL structure.** Webflow slugs default to the page name with spaces converted to hyphens. That's fine for most pages. The place it breaks down is CMS collections: if you name a blog post "5 Things You Should Know About Webflow CMS in 2026 (Updated)", the auto-generated slug is a mess. Set slugs manually in the CMS item before publishing. Keep them short, keyword-forward, and don't include dates on content you plan to update.

**Title tag formula that works for service pages:** `[Primary Keyword] — [Differentiator] | Brand`. Example: `Webflow Developer for SaaS — Senior Freelancer | Mahade Walid`. That's 57 characters, front-loaded with the keyword, differentiated, and branded.

## Schema markup in Webflow (brief overview)

Schema markup tells search engines what your content means, not just what it says. Webflow doesn't generate it automatically — you add it via custom code embeds or a CMS-driven script tag. The page types that benefit most: service pages (Service schema), blog posts (BlogPosting), FAQ sections (FAQPage), testimonial pages (Review), and case studies (CreativeWork).

The fastest way to add schema in Webflow is a `<script type="application/ld+json">` block in the Page Settings → Custom Code → Head section. For CMS-driven pages, you can reference CMS fields in the schema using Webflow's template syntax. For a full walkthrough of every schema type and how to wire each one up in Webflow, the dedicated post on [Webflow schema markup](/blog/webflow-schema-markup) covers it with copy-paste templates.

The one I add to every project without exception: FAQPage schema on any page with an FAQ section. It's 20 minutes of work and has a meaningful chance of earning an FAQ rich result in SERPs — expanding your listing's vertical footprint without earning a new position.

## XML sitemap and robots.txt

Webflow auto-generates an XML sitemap at `/sitemap.xml` when you enable it in Site Settings → SEO → Sitemap. Enable it. Once your custom domain is live and connected, submit the sitemap URL to Google Search Console via the Sitemaps report.

Two things that break Webflow sitemaps that aren't obvious: first, CMS collection pages only appear in the sitemap if the collection is set to "Create a page for each item." If you've built a CMS collection purely for reference data (say, a team members collection that doesn't have its own template page), those items won't appear, which is correct. Second, pages with "Exclude from search" enabled in Page Settings won't appear in the sitemap — verify that setting is off for everything you want indexed.

Webflow's default `robots.txt` allows all bots and points to the sitemap. You can't edit it directly. If you need to block a specific directory (like a client preview path), use a meta robots tag on individual pages rather than trying to modify robots.txt. For a detailed breakdown of sitemap edge cases, canonical setup, and how to handle CMS pagination, the [Webflow sitemap and robots guide](/blog/webflow-sitemap-robots-canonical) covers the specifics.

## Canonicals and duplicate content

Canonicals tell search engines which URL is the "real" version of a page when multiple URLs could serve the same content. Webflow sets a canonical automatically based on the page's published URL, which is usually correct — but there are three scenarios where it goes wrong.

<Callout type="warning">
The most common canonical mistake I see on Webflow sites: forgetting to disable indexing on the `.webflow.io` staging subdomain after connecting a custom domain. The staging subdomain stays live and accessible, Google crawls it, and both versions end up in the index — the canonical on the staging domain points to itself, not the live domain. Fix this immediately: Site Settings → SEO → check "Disable Webflow subdomain indexing."
</Callout>

**Scenario 1: CMS collection pagination.** If your blog listing at `/blog` paginates to `/blog?page=2`, Webflow doesn't automatically add a canonical on the paginated page pointing back to `/blog`. Add a canonical tag manually via custom code on those paginated views, or use rel=prev/next if you prefer the pagination signal approach.

**Scenario 2: UTM parameters.** Traffic with UTM parameters creates new URLs that can get indexed. Canonical tags resolve this — Webflow's automatic canonical covers it as long as you haven't done anything to override it in custom code.

**Scenario 3: HTTP vs HTTPS vs www vs non-www.** Pick one canonical form — `https://yourdomain.com` or `https://www.yourdomain.com` — and make sure all redirects enforce it. Set this in Webflow's domain settings by making one the primary domain and 301-redirecting the others.

## Core Web Vitals

Google's Core Web Vitals — LCP (Largest Contentful Paint), INP (Interaction to Next Paint, replacing FID), and CLS (Cumulative Layout Shift) — are a ranking signal. Not the biggest one, but failing them when competitors pass is unnecessary lost ground.

Webflow's CDN and clean HTML output mean you start from a better baseline than most WordPress sites. Where Webflow builds typically underperform:

**LCP failures** almost always come from an unoptimized hero image — large file, wrong format, no preload hint. The hero image is almost always the LCP element. Setting it as a background image in Webflow's designer (instead of an `<img>` tag) also removes the ability to add a `fetchpriority="high"` attribute, which is the fastest single fix for LCP. Use an `<img>` element for the hero, add a custom attribute of `fetchpriority` with value `high`, and make sure the file is WebP under 150KB.

**CLS failures** in Webflow almost always come from fonts loading late and causing layout shifts (set `font-display: swap` in your custom font CSS), or from images without explicit width and height attributes. Every `<img>` tag should have width and height set so the browser reserves space before the image loads.

The deeper fixes — lazy loading, interaction weight, third-party script management — are in the dedicated [Core Web Vitals fixes for Webflow](/blog/webflow-core-web-vitals) guide. If your site is slow and you're not sure why, start with [why your Webflow site is slow](/blog/webflow-site-slow) before working through the CWV post.

## Image optimization for SEO

Images are the single most reliable quick win on Webflow SEO audits. Most sites I look at have at least one of these three problems: files are too large, alt text is missing or auto-generated from filenames, or images are served as PNG when they should be WebP.

**File size.** The target for any body image is under 200KB in WebP. Hero images can go up to 300KB if they're full-width. Background images on sections should be under 150KB. Webflow's image optimization in the Designer is decent but not aggressive — upload optimized files rather than relying on Webflow's compression pipeline.

**Alt text.** Every `<img>` needs a descriptive alt attribute. In Webflow CMS, that means adding an Image field for each image with a dedicated alt text field alongside it — not relying on the image file name. For decorative images (icons, dividers, background textures served as img tags), set alt to an empty string (`alt=""`), not to the filename.

**Format and loading.** Webflow will serve WebP in modern browsers when you upload WebP files. For CMS images uploaded by editors, add a note in your content guide reminding them to export WebP. For images below the fold, Webflow's lazy loading handles most cases — but verify it's on in the Designer for each image element. The full [Webflow image optimization guide](/blog/webflow-image-optimization) covers srcset, responsive sizes, and the cases where Webflow's built-in optimization falls short.

**Filename SEO.** Before uploading, rename image files to be descriptive: `webflow-cms-collection-screenshot.webp` is better than `image-47.webp`. It's a minor signal but a free one.

## Internal linking structure

Internal links distribute authority from high-traffic pages to the pages you most want to rank. Most Webflow sites I audit have a flat internal link structure — every page links to the homepage and that's about it. That's leaving ranking equity on the table.

The principle is simple: your highest-authority pages (typically the homepage and any page with backlinks pointing to it) should link explicitly to the pages you want to rank. Blog posts should link to relevant service pages. Service pages should link to case studies that prove the service. Case studies should link back to the service page and to related blog content.

**In Webflow specifically:** the navbar and footer are global components, so links there pass equity to every page they point to equally. Don't put 30 links in your footer — it dilutes the equity signal. Keep footer links to the pages that matter most for search: key service pages, the blog index, and contact.

**For CMS-driven content:** use reference or multi-reference fields to surface related posts at the bottom of every blog post. This creates systematic internal linking that grows with your content. A blog with 50 posts and zero internal links between them is 50 separate pages fighting for attention independently. Wire them together.

**Anchor text.** Use descriptive anchor text, not "click here" or "read more." When I link to a guide about Webflow CMS, the anchor text is "Webflow CMS structure" — the keyword I want that destination page to rank for. Webflow doesn't constrain you here — the discipline is entirely on the writer.

## Webflow CMS SEO: collection URLs, dynamic meta

CMS-driven pages require extra attention because one template controls the SEO for potentially hundreds of pages. Getting the template right multiplies across your entire collection.

**Collection URL structure.** In Webflow, collection URLs follow the pattern `/collection-name/item-slug`. For blog posts, that's `/blog/post-slug`. For case studies, `/work/project-slug`. Choose collection names that are keyword-friendly and don't change them after indexing — changing the collection URL path is a breaking change that requires 301 redirects on every item. I've written about how to plan this upfront in the [Webflow CMS structure guide](/blog/webflow-cms-structure).

**Dynamic meta fields.** In the CMS Template page settings, set the SEO Title and Meta Description to pull from CMS fields, not hardcoded text. Create a dedicated "SEO Title" text field and "Meta Description" text field in your collection schema — separate from the post title — so editors can write optimized meta independently from the headline. If those fields are empty, fall back to the post title (most collections support conditional logic in the template meta binding).

**Open Graph for CMS.** Set the OG image field to the CMS item's hero image. Set OG title and description to the same dedicated meta fields. Social shares of CMS pages with a blank OG image — or Webflow's default thumbnail — are a missed opportunity.

For a full walkthrough of CMS collection setup from an SEO perspective, the [Webflow blog setup guide](/blog/webflow-blog-setup) covers the blog-specific decisions in detail.

**Slug uniqueness.** Webflow enforces unique slugs within a collection but not across collections. A blog post and a case study can share the same slug at different paths — that's fine. What causes problems is changing a slug after indexing. If you must change a slug, add a 301 redirect from the old URL to the new one in Site Settings → Redirects immediately.

## Localization and hreflang (if applicable)

If your site serves a single language to a single country, skip this section — it doesn't apply.

If you're running a multilingual Webflow site, or a site targeting the same language in multiple countries (US English vs. UK English vs. AU English), hreflang attributes tell Google which version to serve to which audience. Webflow doesn't have native hreflang support — you add the tags manually via custom code in the `<head>` of each localized template.

The implementation looks like this: for each page, include one `<link rel="alternate" hreflang="en-US" href="https://yourdomain.com/page/">` tag per language/region variant, plus a `<link rel="alternate" hreflang="x-default" href="https://yourdomain.com/page/">` for the fallback. Every page must list all its own variants, including itself — a common mistake is listing other variants but not self-referencing.

<Callout type="note">
Do not implement hreflang unless you actually have separate, translated content for each locale. Adding hreflang tags on an English-only site targeting multiple countries (US, UK, CA) without locale-specific content does more harm than good — Google will either ignore the tags or get confused by the conflicting signals. Use hreflang only when you have genuinely different content per locale.
</Callout>

For most Webflow marketing sites, localization means one thing: if your Webflow Localization feature is enabled (released late 2023), use it for translated copy and let Webflow handle the alternate link tags. If you're doing localization manually via separate CMS collections, you'll need to wire the hreflang tags yourself.

## Launch checklist: the 12 things to verify before shipping

This is the list I run through on every site in the 48 hours before publishing to the live domain. I've missed items on this list before. Every time I did, it cost time after launch to fix.

1. **Custom domain is connected and set as primary** — Webflow dashboard shows green checkmark; www and non-www both redirect to the primary form.
2. **SSL certificate is active** — `https://yourdomain.com` loads without a security warning; HTTP redirects to HTTPS.
3. **Webflow subdomain indexing is disabled** — Site Settings → SEO → "Disable Webflow subdomain indexing" is checked.
4. **All pages have unique title tags and meta descriptions** — Open each page in Page Settings, check the SEO tab. No two pages should share a title.
5. **H1 is present and unique on every page** — Check each page in the Designer; filter by "H1" if using Webflow's element tree.
6. **Canonical tags point to the live domain** — View source on the homepage and one CMS item; confirm canonical is `https://yourdomain.com/` not the staging subdomain.
7. **Sitemap is live and includes key pages** — Visit `/sitemap.xml`, count entries, verify CMS collection pages appear.
8. **Search Console is verified** — Add the property, verify via HTML tag in head, submit the sitemap URL.
9. **robots.txt is not blocking crawling** — Visit `/robots.txt`, confirm there's no `Disallow: /` line blocking Googlebot.
10. **All redirects from the old site are in place** — If migrating from an existing URL structure, every old URL that had backlinks or indexed traffic should 301 to the correct new URL.
11. **Core Web Vitals pass on the homepage** — Run PageSpeed Insights on the live domain (not staging). LCP under 2.5s, CLS under 0.1.
12. **Schema markup validates** — Paste the homepage URL and one blog post URL into Google's Rich Results Test. Fix any errors before launch, not after.

If you're migrating from another CMS, item 10 is the one that bites hardest. A redirect map that misses high-authority pages can drop organic traffic 30–40% even on a technically cleaner site. Build the redirect map before launch, not after you notice the traffic drop.

## Post-launch: Search Console, monitoring, and the first 30 days

Launching the site is not the end of the SEO work. The first 30 days set the baseline that all future measurements are compared against. Here's what I do and recommend clients do in that window.

| Week | Actions | What you're watching for |
|---|---|---|
| Week 1 | Verify Search Console ownership, submit sitemap, run Coverage report | Crawl errors, excluded pages that should be indexed, any "Page with redirect" or "Soft 404" issues |
| Week 2 | Check Coverage daily, run Rich Results Test on key templates, set up position tracking for 10–15 target keywords | New indexing errors, schema errors in Search Console's Enhancements report, any manual actions |
| Week 3 | Pull first performance data (impressions, clicks, average position for indexed pages), review Core Web Vitals report in Search Console | Whether Google is associating your target keywords with your pages; any CWV failures flagged in the report |
| Week 4 | Compare indexed page count against expected, review internal linking from any new blog posts published, update any page titles that got 0 impressions in the first 3 weeks | Title mismatch (Google rewrites titles when they're misaligned with content), missing indexing, pages ranking for wrong keywords |

The metric that matters most in the first 30 days is not rankings — it's indexed page count and crawl coverage. If Google isn't indexing your pages, no amount of on-page optimization matters. Watch the Coverage report first, keyword positions second.

**Setting up position tracking.** Use Google Search Console's Performance report as your primary position tracker — it's free, it's accurate for your domain specifically, and it integrates with your site's actual data. Third-party rank trackers (Ahrefs, SEMrush, Moz) are useful for competitive intelligence but not necessary in the first 30 days.

**First content cadence.** The fastest way to build crawl authority on a new Webflow site is to publish consistently in the first 90 days. Two blog posts per month is enough to signal that the site is active and being updated. More is better, but thin or low-quality posts hurt — don't publish for publication's sake.

**Backlinks.** One thing this checklist intentionally does not cover in depth: link building. It's outside the scope of what you control inside Webflow, and it's a separate discipline from technical SEO. What's worth knowing: the technical foundation you build with this checklist is what makes link building work. A fast, cleanly structured site with correct canonicals and schema converts earned backlinks into ranking improvements efficiently. A slow, misconfigured site absorbs backlinks and shows weak results because there are too many other friction points in the ranking equation.

The 80/20 of Webflow SEO is this: ship fast, write real content, and don't break the basics. Every item in the launch checklist is a "don't break the basics" check. The Core Web Vitals work is the "ship fast" work. The content cadence and internal linking are the "write real content" work. The rest of the checklist — schema, hreflang, CMS meta templates, canonical edge cases — is the remaining 20% that compounds over time once the foundation is solid.
